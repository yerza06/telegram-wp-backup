# Техническое Задание
## Telegram-бот для резервного копирования WordPress сайта

---

## 1. Общее описание

Локальный Telegram-бот для управления резервным копированием **одного** WordPress-сайта и мониторинга дискового пространства. Бот устанавливается **на тот же сервер**, где работает WordPress, и выполняет все операции (`tar`, `mysqldump`, `mysql`, `df -h`, `cp`) локально через дочерние процессы.

**Один экземпляр бота = один сайт = один сервер.** Если сайтов несколько — поднимается несколько независимых экземпляров бота, каждый со своим `.env` и своей БД.

**Язык интерфейса:** русский.

**Что НЕ нужно делать:**
- SSH-подключения к удалённым серверам (бот всегда локальный).
- Управление несколькими сайтами одним ботом. Все параметры сайта (путь, БД, креды MySQL) хранятся в `.env`, а не в БД.
- Автоматическое удаление старых бэкапов (без ретеншена и лимитов).
- Скачивание бэкапов через Telegram. После создания бэкапа бот присылает полный путь к архиву на сервере; пользователь сам копирует его через `scp`/`rsync` (например, на Google Drive вручную).
- Интеграция с Google Drive / S3 / любым облаком.
- Выбор типа бэкапа (только файлы / только БД) — всегда полный (файлы + БД).

## 2. Стек технологий

| Компонент | Технология |
|---|---|
| Язык | Python 3.13+ |
| Менеджер пакетов | uv |
| Telegram-фреймворк | aiogram 3.x |
| ORM | SQLAlchemy 2.x (async) |
| БД | SQLite |
| БД-драйвер | aiosqlite |
| Миграции | Alembic |
| Планировщик | APScheduler (`AsyncIOScheduler`) |
| Конфигурация | pydantic-settings |
| Локальные команды | `asyncio.create_subprocess_exec` (`tar`, `mysqldump`, `mysql`, `df`, `cp`, `rm`, `chown`) |

> **Почему SQLite, а не MySQL/MariaDB.** Объём данных бота копеечный (десятки пользователей, сотни записей о бэкапах за всё время). Бот — единственный писатель, конкурентности по записи нет. SQLite физически изолирован от MySQL-инстанса WordPress, поэтому восстановление WP-БД через `mysql < db.sql` не может задеть данные бота. Деплой проще: один файл рядом с ботом, бэкап = `cp backups.db`. Дополнительных прав/пользователей в прод-MySQL заводить не нужно.

## 3. База данных

В БД хранятся **только** пользователи бота, история бэкапов и расписания. Параметры сайта (путь, имя БД, креды) — в `.env`, потому что бот обслуживает ровно один сайт.

### Таблица `users`
| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER PK | Внутренний ID |
| telegram_id | BIGINT UNIQUE | ID пользователя в Telegram |
| username | VARCHAR | Username в Telegram |
| fullname | VARCHAR | Полное имя пользователя (имя, фамилия) |
| role | ENUM | `admin`, `viewer` |
| is_active | BOOLEAN | Активен ли пользователь |
| created_at | DATETIME | Дата добавления |

### Таблица `backups`
| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER PK | Внутренний ID |
| status | ENUM | `pending`, `running`, `success`, `failed` |
| file_name | VARCHAR | Имя файла архива (`wp-steam-backup_2026-04-09_14-30-09.tar`) |
| backup_path | VARCHAR | Полный путь к файлу бэкапа на сервере |
| size_bytes | BIGINT | Размер архива в байтах |
| has_warnings | BOOLEAN DEFAULT FALSE | Tar вернул код 1 (часть файлов менялась/исчезала во время архивации). Бэкап считается успешным, но требует внимания. См. §5.3 |
| error_message | TEXT | Сообщение об ошибке (если `failed`) |
| triggered_by | BIGINT FK → users.telegram_id (ON DELETE SET NULL) | telegram_id того, кто запустил. `1` = действие superadmin (см. §4). NULL если запущено по расписанию или пользователь удалён |
| is_scheduled | BOOLEAN | Плановый или ручной |
| created_at | DATETIME | Дата создания |
| completed_at | DATETIME | Дата завершения |

### Таблица `restores`
| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER PK | Внутренний ID |
| backup_id | FK → backups (ON DELETE SET NULL) | Из какого бэкапа восстанавливали (NULL если бэкап позже удалили) |
| status | ENUM | `running`, `success`, `rolled_back`, `failed` |
| triggered_by | BIGINT FK → users.telegram_id (ON DELETE SET NULL) | telegram_id того, кто запустил восстановление. `1` = действие superadmin (см. §4) |
| safety_dir | VARCHAR | Путь к защитному snapshot БД (`$SAFETY_DIR/db_current.sql`). NULL если уже удалён |
| error_message | TEXT | Сообщение об ошибке (если `failed` или `rolled_back`) |
| created_at | DATETIME | Дата старта восстановления |
| completed_at | DATETIME | Дата завершения |

> Отдельная таблица `restores` (а не доп. поля в `backups`) — потому что один и тот же бэкап можно восстанавливать несколько раз, и каждое восстановление это самостоятельное событие со своим автором, статусом и временем.

### Таблица `schedules`
| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER PK | Внутренний ID |
| name | VARCHAR NOT NULL | Короткое имя расписания (показывается в меню бота) |
| description | VARCHAR NULL | Свободное описание (для чего нужно, кто завёл) |
| cron_expression | VARCHAR | Cron-строка (`0 2 * * *`) |
| is_active | BOOLEAN | Включено ли расписание |
| created_at | DATETIME | Дата создания |

> Таблиц `sites` и `user_site_permissions` нет — они не нужны при модели «один бот = один сайт».

## 4. Роли и права доступа

| Действие | superadmin | admin | viewer |
|---|---|---|---|
| Добавить/удалить пользователя | ✅ | ❌ | ❌ |
| Изменить роль пользователя | ✅ | ❌ | ❌ |
| Создать бэкап вручную | ✅ | ✅ | ❌ |
| Восстановить из бэкапа | ✅ | ✅ | ❌ |
| Удалить бэкап | ✅ | ✅ | ❌ |
| Просмотр списка бэкапов | ✅ | ✅ | ✅ |
| Просмотр истории восстановлений | ✅ | ✅ | ✅ |
| Управление расписанием | ✅ | ✅ | ❌ |
| Очистить кэш WordPress (`wp cache flush`) | ✅ | ✅ | ❌ |
| Просмотр дискового пространства сервера | ✅ | ✅ | ✅ |

**Superadmin** задаётся через переменную среды `TG__SUPERADMIN` (это реальный telegram_id оператора). Реальный telegram_id в `users` **не записывается** — авторизация superadmin идёт прямой проверкой `from_user.id == settings.tg.superadmin`, без обращения к БД, поэтому он автоматически получает полный доступ и не проходит проверку `is_active`.

Чтобы FK `triggered_by → users.telegram_id` оставался валидным для действий, выполненных superadmin, при старте бота делается **upsert виртуальной строки** в `users`:

```
telegram_id = 1
username    = 'superadmin'
fullname    = 'superadmin'
role        = 'admin'
is_active   = true
```

Все действия superadmin (бэкап, restore) пишутся в `backups.triggered_by = 1` и `restores.triggered_by = 1`. В UI «кто запустил» отображается `fullname` этой виртуальной строки — `superadmin`. Реальный telegram_id оператора в БД не утекает.

**Логика блокировки:**
- `is_active = false` — пользователь заблокирован, бот отвечает «доступ не разрешён» **независимо от роли**.
- `is_active = true` — пользователь работает с ботом согласно своей роли.
- Управлять пользователями (активация, деактивация, смена роли) может **только superadmin**.

## 5. Функциональные требования

### 5.1 Управление пользователями

**Логика регистрации:**
1. Пользователь отправляет `/start`.
2. Если пользователя нет в БД — он автоматически добавляется с `role = viewer` и `is_active = false`.
3. Бот сообщает, что доступ ожидает подтверждения от администратора.
4. Если `is_active = false` — все остальные команды блокируются.
5. Если `is_active = true` — пользователь работает согласно своей роли.

**Команды superadmin для управления пользователями:**
- Просмотр списка всех пользователей (`telegram_id`, `username`, `role`, `is_active`).
- Активировать пользователя.
- Деактивировать пользователя.
- Изменить роль (`admin` / `viewer`).

### 5.2 Просмотр дискового пространства
- Команда `df -h` через `asyncio.create_subprocess_exec`, человекочитаемый вывод в чат.
- Отдельная внутренняя проверка свободного места в `settings.backup.dir` через `shutil.disk_usage` перед каждым бэкапом (см. §5.3).

### 5.3 Резервное копирование

**Формат имени архива:**
```
wp-<settings.site.name>-backup_<YYYY-MM-DD_HH-mm-SS>.tar.zst
```
Пример: `wp-steam-backup_2026-04-09_14-30-09.tar.zst`

Сжатие — **zstd** (`tar --zstd`). Выбран из-за лучшего соотношения степень/скорость по сравнению с gzip (~10–20% компактнее, в 3–5× быстрее на сжатии, в ~2× быстрее на распаковке). Бинарь `zstd` обязателен на сервере (см. §7).

**Структура архива (внутри один tar, не вложенные архивы):**
```
wp-steam-backup_2026-04-09_14-30-09.tar.zst
├── wordpress/          # Обычная папка с файлами WordPress (не вложенный архив)
│   ├── wp-content/
│   ├── wp-config.php
│   └── ...
└── database/           # Обычная папка с дампом БД
    └── db.sql
```

> Это **один** `.tar.zst`, внутри которого две папки верхнего уровня. Не «архив в архиве». Распаковка одной командой: `tar --zstd -xf <archive> -C <dest>`. Просмотр содержимого: `tar --zstd -tvf <archive>`.

**Процесс создания бэкапа (всё локально, без SSH):**
1. **Конкурентность.** Проверить, что для сайта уже не выполняется бэкап или восстановление (см. §5.8). Если выполняется — отказать с понятным сообщением.
2. **Свободное место.** `shutil.disk_usage(settings.backup.dir).free` ≥ `settings.backup.free_space_mb`. Если меньше — прервать с ошибкой и уведомлением.
3. Создать каталог `{settings.backup.dir}/{settings.site.name}/`, если не существует.
4. Создать временную папку сборки внутри `BACKUP__TMP_DIR`:
   `TMP_DIR = tempfile.mkdtemp(prefix="wp-backup-{site.name}-", dir=settings.backup.tmp_dir)`.
   `BACKUP__TMP_DIR` должен лежать на той же файловой системе, что `WP_PATH` и `BACKUP__DIR` — иначе финальный `mv` архива будет деградировать в `cp+rm`. Это проверяется на старте бота через `os.stat(...).st_dev` (см. §6).
5. Внутри `TMP_DIR`:
   - `mkdir database`.
   - Сделать дамп БД: `mysqldump --single-transaction --quick -u{site.db_user} {site.db_name} > $TMP_DIR/database/db.sql`. Пароль передаётся через переменную окружения дочернего процесса `MYSQL_PWD`, **не** через `-p{pass}` (см. §7).
6. Собрать архив **напрямую** из `WP_PATH` без промежуточного `cp -a` (один проход чтения, пик диска ≈ 1× размера сайта вместо 2×). Tar поддерживает несколько `-C`. Флаг `--ignore-failed-read` обязателен: WordPress может удалять кэш/сессии между `readdir` и `open`, без этого флага tar упадёт с кодом 2 даже на штатной ситуации.
   ```
   tar --zstd --ignore-failed-read -cf {backup.dir}/{site.name}/wp-{site.name}-backup_<timestamp>.tar.zst \
       -C $TMP_DIR database \
       -C $(dirname $WP_PATH) $(basename $WP_PATH)
   ```
   Внутри архива папка с файлами WP получит имя `basename($WP_PATH)` (например, `wordpress` или `steam`). Это нужно учитывать при распаковке во время restore (см. §5.4).
7. **Обработка кода возврата tar:**
   - `0` → `status = success`, `has_warnings = false`. Штатный путь.
   - `1` → `status = success`, `has_warnings = true`. Часть файлов менялась/исчезала во время архивации — это норма для живого WordPress (см. disclaimer ниже). Архив валиден, в уведомлении добавляется пометка про warnings.
   - `2` или любой другой ненулевой → `status = failed`, `error_message` = stderr tar. Архив удаляется (если успел создаться).
8. Удалить временную папку: `shutil.rmtree(TMP_DIR, ignore_errors=True)`. Удаляется **только** созданная нами подпапка внутри `BACKUP__TMP_DIR` — никогда не «всё содержимое временной директории».
9. Сохранить в БД: `size_bytes`, `backup_path`, `file_name`, `status`, `has_warnings`, `completed_at`, `triggered_by`.
10. Отправить уведомление всем активным пользователям: размер, **полный путь к архиву** (для последующего ручного `scp`/`rsync`). Если `has_warnings = true` — добавить предупреждение.

> **Hot-backup и его компромиссы.** Бэкап выполняется на живом сайте **без maintenance mode** — это сознательный компромисс в пользу нулевого даунтайма. Последствия, которые оператор должен знать:
> - **Рассинхрон файлов и БД.** `mysqldump --single-transaction` фиксирует БД на момент T1, tar читает файлы между T1 и T2 (минуты на 20 GB сайта). Запись о новом посте может оказаться в дампе БД, а файл медиа — нет (пользователь загрузил его после T1). После restore это будет битая ссылка в админке WP.
> - **Частично записанные загрузки.** Если пользователь грузит файл через wp-admin в момент работы tar — в архив попадёт обрезанный JPG/PDF.
> - **Tar код 1 — это норма.** WordPress активно пишет в `wp-content/cache/`, `wp-content/uploads/`, sessions. Файлы постоянно удаляются/переименовываются между `readdir` и `open`. Код 1 означает «архив создан, но некоторые файлы были пропущены» — для живого WP это штатная ситуация, бот трактует её как success-with-warnings.
>
> Для критичных моментов (миграция, перед опасным обновлением плагинов, перед деплоем) оператор может вручную включить maintenance mode через WP-CLI на сервере **до** запуска бэкапа через бота — отдельной кнопки «бэкап с maintenance mode» в первой версии бота нет.

> **Важно про `BACKUP__TMP_DIR`.** Это новая переменная в `.env` (см. §6). Она заменяет дефолтный `/tmp` потому, что:
> 1. `/tmp` на многих системах смонтирован как `tmpfs` (RAM) или отдельный раздел — `mv` через границу ФС деградирует в `cp+rm` (медленно, не атомарно).
> 2. Для сайта в 20 GB `/tmp` обычно слишком маленький.
>
> Папка одна и для backup, и для restore — это безопасно, потому что обе операции взаимоисключающи через лок из §5.8.

**Имя файла архива:** `wp-<site.name>-backup_<YYYY-MM-DD_HH-mm-SS>.tar.zst`. Пример: `wp-steam-backup_2026-04-09_14-30-09.tar.zst`.

**Команды бота:**
- Создать бэкап.
- Список бэкапов (с датой, размером, статусом).
- Количество бэкапов.
- Удалить бэкап (выбор из списка → подтверждение → удаление файла + записи в БД).

> **Ретеншен (автоматическое удаление старых бэкапов) не реализуется** — это явное требование §1. Оператор сам забирает архивы через `scp`/`rsync` на свою машину/Google Drive и удаляет их с сервера вручную через команду «Удалить бэкап» в боте.

### 5.4 Восстановление (с rollback через mv-подмену)

**Ключевая идея.** Файловый snapshot текущего `WP_PATH` делается **не через `cp -a`**, а через `mv WP_PATH WP_PATH.old-<restore_id>`. Это:
- мгновенно (`rename(2)` ядра, не зависит от размера сайта),
- занимает 0 байт дополнительного места на момент создания,
- атомарно — между «есть старые файлы» и «есть новые файлы» нет состояния «папка наполовину перезаписана».

В `$SAFETY_DIR` хранится **только дамп текущей БД** (`db_current.sql`) — он маленький (десятки МБ).

Атомарность `mv` гарантирована **только** если источник и назначение на одной файловой системе. Поэтому `BACKUP__TMP_DIR` обязан лежать на том же диске, что и `WP_PATH` — это валидируется на старте бота (см. §6). Если ФС разные — `mv` деградирует в `cp+rm`, операция всё равно работает, но становится не атомарной и в разы медленнее. Бот пишет об этом warning в лог при старте.

**Процесс:**
1. Выбрать бэкап из списка.
2. Подтверждение (⚠️ это перезапишет текущие файлы и БД).
3. Проверка конкурентности (см. §5.8).
4. **Создать запись в `restores`** со `status = running`, `backup_id`, `triggered_by` (telegram_id того, кто запустил, либо `1` для superadmin).
5. **Snapshot БД** (только БД, не файлы):
   - `SAFETY_DIR = tempfile.mkdtemp(prefix="wp-rollback-{site.name}-", dir=settings.backup.tmp_dir)`.
   - `mysqldump --single-transaction --quick > $SAFETY_DIR/db_current.sql` (`MYSQL_PWD` через env).
   - Сохранить путь `$SAFETY_DIR` в поле `restores.safety_dir`.
6. **Распаковать архив во временную папку** на том же диске:
   - `TMP_DIR = tempfile.mkdtemp(prefix="wp-restore-{site.name}-", dir=settings.backup.tmp_dir)`.
   - `tar --zstd -xf <backup>.tar.zst -C $TMP_DIR`.
   - **Если распаковка упала** (например, не хватило места на диске): `shutil.rmtree(TMP_DIR, ignore_errors=True)` + `shutil.rmtree(SAFETY_DIR, ignore_errors=True)` + `restores.status = failed`. **`WP_PATH` не тронут — сайт продолжает работать.** Никакого отката не нужно, потому что подмены ещё не было.
7. **Включить maintenance mode WordPress** (обязательно, не опционально):
   ```
   sudo -u www-data wp maintenance-mode activate --path={site.wp_path}
   ```
   Это закрывает окно неконсистентности «новые файлы — старая БД» и предотвращает потерю пользовательских записей в БД во время restore. Если WP-CLI не установлен — restore запрещён, бот возвращает ошибку с инструкцией по установке.
8. **Атомарная подмена файлов через два `rename(2)`:**
   ```
   mv $WP_PATH ${WP_PATH}.old-<restore_id>          # бывшие файлы → snapshot
   mv $TMP_DIR/<basename($WP_PATH)> $WP_PATH        # новые файлы → на место
   ```
   Окно «нет сайта» — микросекунды между двумя `rename` (на одной ФС).
9. **Восстановление владельца:** `chown -R www-data:www-data {site.wp_path}`. Права файлов/каталогов **не** правим через `find ... chmod 755/644` — tar восстанавливает оригинальные права из архива, а массовый `chmod` затирает кастомные (например, `wp-config.php` обычно `640`).
10. **Восстановление БД:** `mysql {site.db_name} < $TMP_DIR/database/db.sql` (`MYSQL_PWD` через env).
11. **Выключить maintenance mode:**
    ```
    sudo -u www-data wp maintenance-mode deactivate --path={site.wp_path}
    ```
12. **Если любой из шагов 8–11 упал — выполнить откат:**
    - `rm -rf $WP_PATH` (выкинуть распакованные новые файлы).
    - `mv ${WP_PATH}.old-<restore_id> $WP_PATH` (вернуть старые файлы — снова `rename(2)`, мгновенно).
    - Восстановить БД из `$SAFETY_DIR/db_current.sql`.
    - `sudo -u www-data wp maintenance-mode deactivate --path={site.wp_path}`.
    - Удалить `$SAFETY_DIR` и `$TMP_DIR`.
    - Обновить запись в `restores`: `status = rolled_back`, `error_message = "<причина>"`, `completed_at = now()`.
    - Уведомить пользователя: «Восстановление не удалось, выполнен откат к предыдущему состоянию».
13. **Если всё успешно:**
    - Удалить `${WP_PATH}.old-<restore_id>`, `$TMP_DIR` и `$SAFETY_DIR`.
    - Обновить запись в `restores`: `status = success`, `safety_dir = NULL`, `completed_at = now()`.
14. **Если откат тоже упал** — оставить `$SAFETY_DIR` и `${WP_PATH}.old-<restore_id>` на диске, обновить `restores`: `status = failed`, `error_message`, `completed_at`. В уведомлении указать полные пути к обоим артефактам для ручного восстановления.
15. Уведомление о результате (успех / откат / критическая ошибка).

> Защитный snapshot БД (`$SAFETY_DIR/db_current.sql`) и файловый snapshot (`${WP_PATH}.old-<restore_id>`) **не удаляются**, если откат тоже упал — оператор должен иметь возможность ручного восстановления. В уведомлении и в `restores.safety_dir` указываются полные пути.

**Команды бота для просмотра истории:**
- Список бэкапов — для каждого: дата, размер, статус, **кто создал** (`triggered_by`).
- История восстановлений — для каждого: дата, **из какого бэкапа** (`backup_id` → `file_name`), **кто запустил** (`triggered_by`), статус (success/rolled_back/failed).
- Карточка бэкапа — показывает все восстановления, которые были сделаны из этого бэкапа.

### 5.5 Очистка кэша WordPress

После восстановления, обновления плагинов или просто по требованию администратора иногда нужно сбросить объектный кэш WordPress. Доступно ролям **superadmin** и **admin**.

**Команда:**
```
sudo -u www-data wp cache flush --path={site.wp_path}
```

**Замечания по реализации:**
- Используется WP-CLI (`wp`). На сервере должен быть установлен бинарь `wp` (`/usr/local/bin/wp` обычно). Если его нет — бот должен дать понятную ошибку: «WP-CLI не найден, установите по инструкции https://wp-cli.org/».
- WP-CLI **отказывается работать от root** без флага `--allow-root`. Поскольку бот в первой версии запускается как root (см. §7), запускаем команду от имени `www-data` через `sudo -u www-data`. Это и безопаснее (PHP-код плагинов исполняется под тем же uid, что и в проде), и снимает необходимость в `--allow-root`.
- Флаг `--path={site.wp_path}` нужен, потому что бот не делает `chdir` в каталог сайта.
- Команда выполняется через `asyncio.create_subprocess_exec`. Если код возврата ≠ 0 — ошибка передаётся пользователю как есть (вывод stderr).
- Это **не** долгая операция, но если идёт бэкап или восстановление — лучше отказать, потому что состояние сайта в этот момент непредсказуемо. Реализация: проверить глобальный `asyncio.Lock` из §5.8, и если занято — отказать пользователю с понятным сообщением.

**Команды бота:**
- Очистить кэш WordPress (одна кнопка → подтверждение → выполнение → результат в чат).

### 5.6 Расписание (cron)

- На базе APScheduler `AsyncIOScheduler` с триггером `CronTrigger.from_crontab(...)`.
- При старте бота — подгрузка всех расписаний с `is_active = true` из БД.
- Управление через бота:
  - Добавить расписание. Пресеты: ежедневно 02:00, еженедельно, ежемесячно + «своё выражение».
  - Просмотр активных расписаний.
  - Включить/выключить расписание.
  - Удалить расписание.

### 5.7 Уведомления

Уведомления отправляются **всем активным пользователям** (`is_active = true`) плюс superadmin. Поскольку бот обслуживает один сайт — фильтрация по доступу к сайту не нужна.

События:
- ✅ Успешный бэкап (размер, время, путь к архиву).
- ⚠️ Бэкап создан с warnings: tar вернул код 1, часть файлов менялась/исчезала во время архивации (см. §5.3). Архив валиден, но требует внимания.
- ❌ Ошибка бэкапа (текст ошибки).
- ✅ Успешное восстановление.
- ⚠️ Восстановление не удалось — выполнен откат.
- ❌ Критическая ошибка восстановления — откат тоже не удался (с путём к `$SAFETY_DIR`).
- 🚨 **Критическая ошибка после краша бота:** обнаружено зависшее восстановление #<id>. Состояние сайта непредсказуемо, требуется ручная проверка. В сообщении — пути к `$SAFETY_DIR` и `${WP_PATH}.old-<id>` (см. §5.8.1).
- ⏰ Плановый бэкап запущен.
- ⚠️ Бэкап отклонён: недостаточно свободного места.

### 5.8 Конкурентность

**Запрещено параллельное выполнение операций над сайтом.** Это касается:
- двух одновременных бэкапов (ручной + плановый, или два ручных);
- бэкапа и восстановления одновременно;
- двух восстановлений одновременно.

**Реализация (двухуровневая):**
1. **In-memory:** один глобальный `asyncio.Lock` на процесс — мгновенная блокировка в рамках запущенного бота.
2. **БД:** перед стартом операции проверяется, нет ли в `backups` записи со `status = 'running'`. Это страхует от потерянных in-memory локов после аварийного перезапуска.

Если операция занята — отказ с сообщением: «Уже выполняется операция (запущена в `{created_at}` пользователем `{username}`)».

### 5.8.1 Восстановление после краша бота (orphan recovery)

Двухуровневый лок из §5.8 не спасает от одного сценария: бот упал mid-операции (OOM, `kill -9`, ребут сервера, обрыв питания). В этом случае в БД остаётся запись со `status = 'running'`, и при следующем старте проверка из §5.8 начнёт отклонять **все** новые операции с сообщением «уже выполняется» — хотя процесс давно мёртв.

Поэтому при старте бота, **до** запуска APScheduler и перед началом Telegram polling, выполняется orphan-cleanup. Семантика разная для backup и restore, потому что у них разный риск потери данных.

**Шаг 1. Backups в статусе `running`.**

```
SELECT id, backup_path, created_at FROM backups WHERE status = 'running'
```

Для каждой найденной записи:
1. Найти и удалить временный подкаталог сборки: `glob(f"{settings.backup.tmp_dir}/wp-backup-{settings.site.name}-*")` — все совпадения `shutil.rmtree(..., ignore_errors=True)`. Это безопасно, потому что через `asyncio.Lock` параллельных бэкапов быть не может, а orphan cleanup работает **до** старта polling — значит новых бэкапов ещё нет.
2. Если `backup_path` задан и файл существует — удалить его (`os.remove`). Это частично записанный архив, бесполезный.
3. `UPDATE backups SET status='failed', error_message='process crashed or interrupted (recovered at startup)', completed_at=NOW() WHERE id=...`.

Для backup-операций потери данных нет: все артефакты временные, исходный `WP_PATH` tar только **читал**, не трогал.

**Шаг 2. Restores в статусе `running`.**

```
SELECT id, backup_id, safety_dir FROM restores WHERE status = 'running'
```

Здесь **принципиально другая логика**: в момент краша `WP_PATH` мог быть в любом из состояний (оригинальные файлы, `${WP_PATH}.old-<id>`, распакованные новые файлы, частичный rollback). Состояние БД тоже непредсказуемо. Бот **не** может автоматически определить, что правильно.

Для каждой найденной записи:
1. Удалить только `wp-restore-{site.name}-*` подкаталог в `BACKUP__TMP_DIR` (распакованный архив — всегда восстановим из исходного файла бэкапа, это безопасно).
2. **НЕ трогать** `${WP_PATH}.old-<restore_id>` и `safety_dir` — там может лежать единственная копия исходных данных сайта и БД. Их удаляет **только** оператор вручную после ручной проверки.
3. `UPDATE restores SET status='failed', error_message='process crashed or interrupted (recovered at startup). Manual inspection required.', completed_at=NOW() WHERE id=...`.
4. Отправить **критическое** уведомление всем активным пользователям + superadmin (событие 🚨 из §5.7) со всеми путями:
   ```
   🚨 Бот пережил краш во время восстановления #<id>.
   Состояние сайта: непредсказуемо.
   Проверьте вручную:
   - текущее состояние: {WP_PATH}
   - защитный snapshot БД: {safety_dir}/db_current.sql
   - защитный snapshot файлов: {WP_PATH}.old-<id> (если существует)
   Если сайт работает корректно — удалите оба защитных артефакта вручную.
   Если сайт сломан — восстановите через `mysql < db_current.sql` и `mv {WP_PATH}.old-<id> {WP_PATH}`.
   ```

**Шаг 3.** После cleanup — обычный старт (загрузка расписаний из §5.6, запуск polling aiogram).

> Orphan cleanup выполняется **синхронно** до старта polling — если в момент старта бот не может подключиться к БД или не может записать обновление в `restores`, он отказывается стартовать с явной ошибкой. Полузапущенное состояние недопустимо.

## 6. Переменные среды (`.env`)

Переменные сгруппированы по префиксам с разделителем `__` (двойное подчёркивание). pydantic-settings раскладывает их в **вложенные модели** автоматически (`env_nested_delimiter="__"`):

```python
# bot/core/config.py (эскиз)
from pydantic import BaseModel
from pydantic_settings import BaseSettings, SettingsConfigDict

class TelegramSettings(BaseModel):
    bot_token: str
    superadmin: int
    database_url: str

class SiteSettings(BaseModel):
    name: str
    wp_path: str
    db_name: str
    db_user: str
    db_pass: str

class BackupSettings(BaseModel):
    dir: str
    tmp_dir: str
    free_space_mb: int = 40960

class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env",
        env_nested_delimiter="__",
    )
    tg: TelegramSettings
    site: SiteSettings
    backup: BackupSettings

settings = Settings()
# settings.tg.bot_token, settings.site.wp_path, settings.backup.dir, ...
```

```env
# Telegram
TG__BOT_TOKEN=
TG__SUPERADMIN=
TG__DATABASE_URL=sqlite+aiosqlite:///./backups.db

# Параметры сайта
SITE__NAME=steam
SITE__WP_PATH=/var/www/steam
SITE__DB_NAME=steam_wp
SITE__DB_USER=steam_wp
SITE__DB_PASS=

# Бэкапы
# WP_PATH ≈ 20 GB → нужно ≥ 40 GB свободного места под архив + временные файлы
BACKUP__DIR=/backups
# Временная папка для сборки архива и распаковки во время restore.
# ОБЯЗАТЕЛЬНО должна быть на той же файловой системе, что WP_PATH и BACKUP__DIR
# (иначе mv во время restore деградирует в cp+rm — медленно и не атомарно).
# Не используй /tmp — он часто tmpfs/маленький отдельный раздел.
BACKUP__TMP_DIR=/backups/.tmp
BACKUP__FREE_SPACE_MB=40960
```

**Валидация на старте бота.** При запуске бот проверяет:
1. `SITE__WP_PATH` существует и читаемая директория.
2. `BACKUP__DIR` и `BACKUP__TMP_DIR` существуют (создаются если нет) и доступны на запись.
3. `os.stat(BACKUP__TMP_DIR).st_dev == os.stat(SITE__WP_PATH).st_dev == os.stat(BACKUP__DIR).st_dev`. Если ФС разные — warning в лог, бот продолжает работать (`mv` будет деградировать в `cp+rm`).
4. Все нужные бинари в `$PATH`: `mysql`, `mysqldump`, `tar`, `zstd`, `wp`, `sudo`, `chown`, `df` (см. §7).
5. `shutil.disk_usage(BACKUP__DIR).free` ≥ `BACKUP__FREE_SPACE_MB` — иначе warning (бэкап всё равно упадёт при первой попытке).
6. **Orphan recovery (см. §5.8.1).** Пометить все зависшие `running`-записи в `backups` и `restores` как `failed`, удалить временные артефакты бэкапов, **сохранить** защитные артефакты restore (`$SAFETY_DIR`, `${WP_PATH}.old-<id>`), разослать критическое уведомление по каждому orphan-restore. Выполняется синхронно **до** запуска APScheduler и Telegram polling.

Если любой из пп. 1, 2, 4, 6 не выполнен — бот не стартует и пишет понятную ошибку в stderr.

> **Замечание про `DATABASE_URL`.** Положен в `TG__` группу для соответствия исходной структуре, но семантически это БД самого бота, а не Telegram. Если хочется — позже выносится в отдельную группу `DB__URL`.

> `.env` должен иметь права `600` и владельца — пользователя, от имени которого работает бот. В git **никогда** не коммитится.

## 7. Безопасность

Поскольку параметры сайта (включая `SITE__DB_PASS`) переехали в `.env`, шифрование Fernet больше не нужно — защита сводится к правам доступа к файлу `.env`.

**Правила работы с секретами:**
- Файл `.env` — права `600`, владелец = пользователь бота. В git не коммитится.
- `site.db_pass` **никогда** не логируется (даже на уровне `DEBUG`).
- В `mysqldump`/`mysql` пароль передаётся через переменную окружения `MYSQL_PWD` дочернего процесса:
  ```python
  await asyncio.create_subprocess_exec(
      "mysqldump", "-u", settings.site.db_user, settings.site.db_name,
      stdout=open(dump_path, "wb"),
      env={**os.environ, "MYSQL_PWD": settings.site.db_pass},
  )
  ```
  Передавать пароль через `-p{pass}` **запрещено** — он будет виден в `ps aux`.

### От какого пользователя запускать бота

**Короткий ответ:** да, проще всего запускать бота от **root**, потому что при восстановлении нужно `chown -R www-data:www-data` и доступ на запись в `WP_PATH`, который обычно принадлежит `www-data`.

Возможные варианты, от хорошего к плохому:

1. **Бот от root через systemd-юнит** *(рекомендуется как стартовый вариант)*. Самый простой способ. Бот имеет доступ ко всему, что нужно: чтение/запись `WP_PATH`, `chown`, остановка/старт `nginx`/`apache2`. Минус — широкие права. Митигация: бот живёт в отдельном каталоге (например, `/opt/wp-backup-bot/`) с правами `700`, исходники не редактируются «руками», systemd-юнит ограничен `ProtectSystem=strict`, `NoNewPrivileges=true`, `ReadWritePaths=` только нужные пути.

2. **Бот от непривилегированного пользователя + точечный `sudo`**. Создаётся пользователь `wpbot`, добавляется в `/etc/sudoers.d/wpbot`:
   ```
   wpbot ALL=(root) NOPASSWD: /bin/chown -R www-data\:www-data /var/www/steam, /bin/systemctl stop nginx, /bin/systemctl start nginx
   ```
   Бот вызывает `sudo chown ...` через `subprocess`. Безопаснее, но настройка сложнее, и список разрешённых команд должен совпадать с тем, что бот реально делает.

3. **Бот от `www-data`**. Тоже работает: `www-data` владеет `WP_PATH`, `chown` на свой же uid не нужен, всё пишется напрямую. Но нужно дать `www-data` доступ к `BACKUP_DIR`, к sqlite-файлу и (если потребуется) к `systemctl restart nginx`.

В рамках текущего ТЗ выбираем **вариант 1** (root через systemd) — это упрощает первую версию и убирает целый класс ошибок с правами. Если в будущем захочется ужесточить — переходим на вариант 2 без изменений в коде бота, только настройка `sudoers` + префикс `sudo` в командах сервисного слоя.

### Предусловия деплоя

На сервере должны быть установлены:
- `mysql-client` (`mysqldump`, `mysql`).
- `tar`, `chown`, `df` — стандартные coreutils.
- **`zstd`** — для сжатия архивов (см. §5.3). Установка: `apt install zstd`. Tar 1.31+ умеет сам подхватывать `zstd` через флаг `--zstd`.
- **WP-CLI** (`wp`) — для §5.4 (maintenance mode во время restore) и §5.5 (очистка кэша). Установка: `curl -O https://raw.githubusercontent.com/wp-cli/builds/gh-pages/phar/wp-cli.phar && chmod +x wp-cli.phar && mv wp-cli.phar /usr/local/bin/wp`.
- Пользователь `www-data` (стандартный для nginx/apache на Debian/Ubuntu).
- Python 3.13+ и `uv` для запуска самого бота.

Бот при старте проверяет наличие всех этих бинарей в `$PATH` через `shutil.which(...)` и отказывается запускаться, если хоть одного нет (см. §6).

## 9. Приоритет разработки

1. **Фаза 1** — Ядро: `config` (с валидацией всех параметров сайта из `.env`), `database`, модели, миграции Alembic, auth middleware.
2. **Фаза 2** — Бэкапы: `backup_service` через `subprocess`, `disk_service` (`df -h`), уведомления, конкурентность.
